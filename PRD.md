> **版本**：MVP v1.0
> **日期**：2026-03-17
> **域名**：letswitness.com

---

## 一、产品概述

### 1.1 产品定位

LetsWitness 是一个**预测存证与公众验证平台**。用户可以将公众人物、组织或个人在社交媒体、政论节目、公开演讲等场景中做出的"预测性言论"上传至平台，设定未来的验证条件，由社区共同追踪、投票评判其真实性，并最终见证预测是否兑现。

**一句话描述**：记录预言，等待验证，全民见证。

### 1.2 核心价值

| 维度 | 说明 |
|------|------|
| **存证** | 为公开预测建立不可篡改的时间戳记录 |
| **追踪** | 设定明确的验证条件（时间点/事件触发），系统自动提醒 |
| **公信** | 社区投票机制评估帖子的真实度，过滤虚假或断章取义的内容 |
| **见证** | 当验证条件达成时，社区共同裁定预测结果 |

### 1.3 目标用户

- **核心市场**：欧美用户，优先面向英语用户社区
- **内容贡献者**：关注政治、商业、科技、社会议题等公开预测内容的用户，愿意记录并追踪公众人物或组织的承诺与预言
- **围观验证者**：对公共事件和预测兑现结果感兴趣，愿意参与投票和讨论的普通用户
- **信息核查者**：关注信息真实性，希望通过平台辨别伪造内容、断章取义或缺乏上下文的用户

## 三、详细功能需求

### 3.1 用户系统

#### 3.1.1 注册与登录

| 字段 | 说明 |
|------|------|
| 登录方案 | **方案 B：Supabase Auth + Google Provider** |

| 安全 | 使用 Supabase Auth 管理会话、Token 与 OAuth 流程 |

#### 3.1.2 个人主页

- 显示用户基本信息（头像、用户名、注册时间）
- 我发布的预测帖列表
- 我参与投票的帖子列表
- 统计数据：发帖数、参与投票数

---

### 3.2 帖子系统（核心）

#### 3.2.1 创建预测帖

用户通过表单创建一条预测记录，字段如下：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 标题 | 文本 | ✅ | 简要概括预测内容，≤120 字符 |
| 说明文字 | 富文本 | ✅ | 详细描述预测背景和内容，支持超链接，≤5000 字符 |
| 预测来源 | 文本标签 | ✅ | 做出该预测的人物或组织名称（可搜索已有标签或创建新标签） |
| 媒体附件 | 文件上传 | ❌ | 支持上传图片（JPG/PNG/WebP，≤10MB）、音频（MP3/WAV，≤50MB）、视频（MP4/WebM，≤200MB），最多 5 个附件 |
| 验证事件 | 结构化数据 | ✅ | 至少设定一个验证事件（见 3.3） |
| 标签 | 文本标签 | ❌ | 分类标签，如 #政治、#经济、#科技 等 |

**业务规则**：
- 帖子创建后进入「等待验证」状态
- 帖子创建后，标题、说明文字和媒体附件**不可编辑**（存证属性），但可以追加补充说明
- 每条帖子记录精确的创建时间戳

#### 3.2.2 帖子状态

帖子生命周期包含以下状态：

```
[等待验证] ──→ [验证中] ──→ [已验证：预测成真 ✅]
     Pending      Verifying     Fulfilled
                       │
                       └──→ [已验证：预测落空 ❌]
                              Unfulfilled
                       │
                       └──→ [已验证：部分成真 ⚠️]
                              Partially Fulfilled
                       │
                       └──→ [已过期 / 无法验证]
                              Expired
```

- **等待验证（Pending）**：验证条件尚未触发
- **验证中（Verifying）**：验证条件已触发（到达时间点或关联事件已发生），社区投票裁定中
- **已验证（Resolved）**：社区裁定完成，结果为「预测成真」「预测落空」或「部分成真」
- **已过期（Expired）**：超过验证窗口期且无明确结论

#### 3.2.3 帖子详情页

- 顶部：标题 + 预测来源标签
- 媒体展示区：图片轮播 / 音频播放器 / 视频播放器
- 说明文字区
- 验证事件卡片（见 3.3）
- 真实度投票区（见 3.4）
- 评论区（见 3.5）
- 侧栏 / 底部：帖子元信息（创建时间、发帖用户、状态标签、浏览量）

#### 3.2.4 帖子信息流

- 默认按热度排序（综合浏览量、投票数、评论数）
- 支持筛选：
  - 按状态：等待验证 / 验证中 / 已验证 / 已过期
  - 按分类标签
  - 按预测来源
- 支持搜索：关键词匹配标题和说明文字

### 3.3 验证事件系统

每条预测帖必须关联至少一个验证事件。MVP 支持两种类型：

#### 3.3.1 类型一：时间点验证

| 字段 | 说明 |
|------|------|
| 验证日期 | 用户设定的未来日期（必须晚于帖子创建时间） |
| 验证说明 | 到达该时间点时应验证什么，如 "2027年1月1日前，XX公司市值是否突破万亿" |
| 提醒机制 | 到达日期前 7 天、1 天、当天分别发送站内通知给帖子创建者和参与投票的用户 |

**触发逻辑**：
- 到达设定日期时，帖子自动进入「验证中」状态
- 进入验证中后开放一个 **14 天的验证投票窗口**，社区投票裁定预测结果
- 验证投票窗口结束后，根据投票结果自动更新帖子状态

#### 3.3.2 类型二：事件触发验证

| 字段 | 说明 |
|------|------|
| 事件描述 | 触发验证的事件描述，如 "XX国家大选结果公布" |
| 截止日期 | 事件的最晚等待日期（必填，防止帖子永久挂起） |
| 触发方式 | MVP 版本采用**用户手动触发** — 任意用户可以声明该事件已发生，附上证据链接，经社区投票确认后帖子进入「验证中」 |

**触发逻辑**：
- 任意用户可以提交"事件已发生"的声明 + 证据链接
- 当声明获得 ≥10 个确认投票（或帖子创建者确认）后，帖子进入「验证中」状态
- 进入「验证中」后同样开放 14 天验证投票窗口
- 若截止日期到达但事件未被触发，帖子进入「已过期」状态

### 3.4 投票系统

#### 3.4.1 真实度投票（帖子维度）

**目的**：评估帖子内容的真实度——即帖子所记录的预测是否真实存在，是否值得追踪验证。

**投票规则**：

| 项目 | 说明 |
|------|------|
| 投票资格 | 已注册用户，每个帖子每人限投一票 |
| 投票选项 | 二选一：「值得验证」或「不值得验证」 |
| 展示方式 | **水平进度条（0–100）**，值 = 「值得验证」票数 / 总票数 × 100 |
| 实时更新 | 投票后立即更新进度条 |

**分值含义说明**（显示在 UI 上）：

| 区间 | 含义 |
|------|------|
| 80–100 | 🟢 高真实度 — 帖子内容完全符合事实，值得追踪验证 |
| 50–79 | 🟡 有争议 — 可能缺少上下文，或对预测的描述不够公正 |
| 20–49 | 🟠 低可信度 — 内容可能存在断章取义或误导 |
| 0–19 | 🔴 疑似虚假 — 内容极可能是编造或伪造的 |

**UI 展示**：
- 水平直方图（进度条），左侧 0 右侧 100
- 进度条内显示当前百分比数值
- 进度条下方显示总投票人数
- 根据分值区间显示对应颜色（绿/黄/橙/红）

#### 3.4.2 验证结果投票（验证阶段）

当帖子进入「验证中」状态后，开放验证结果投票：

| 项目 | 说明 |
|------|------|
| 投票选项 | 三选一：「预测成真 ✅」「预测落空 ❌」「部分成真 ⚠️」 |
| 投票时限 | 14 天 |
| 结果判定 | 票数最高的选项为最终结果（需达到最低 20 票） |
| 未达最低票数 | 延长 7 天，若仍未达到则标记为「无法验证」 |

## 五、非功能性需求

### 5.1 性能

| 指标 | 目标 |
|------|------|
| 首页加载时间 | < 2 秒（首次访问） |
| 帖子详情页加载 | < 1.5 秒 |
| 投票响应时间 | < 500ms |
| 媒体上传 | 支持断点续传，200MB 视频上传 < 3 分钟（10Mbps 网络） |

### 5.2 安全

- 所有通信使用 HTTPS
- 文件上传需做类型校验和病毒扫描
- 用户密码使用 bcrypt 加密
- 防止 XSS、CSRF、SQL 注入等常见攻击
- 媒体文件存储于对象存储（如 S3），不直接暴露服务器路径

### 5.3 可用性

- 响应式设计，适配桌面端和移动端浏览器
- 支持主流浏览器：Chrome、Firefox、Safari、Edge 最新两个版本

### 5.4 国际化

- MVP 版本以**英文**为默认和主要语言，优先服务欧美用户
- 界面文案使用 i18n 方案管理，为后续扩展西班牙语、德语、法语等语言预留能力

### 6.3 Supabase 表结构草案

以下为更接近实际落地的 MVP 数据表建议，便于后续直接转为 SQL Migration。

| 表名 | 作用 | 核心字段 | 关键约束 / 索引建议 |
|------|------|----------|---------------------|
| `profiles` | 平台用户资料表，对应 Supabase Auth 用户 | `id`, `username`, `email`, `avatar_url`, `created_at` | `id` 与 `auth.users.id` 一一对应；`username` 唯一；`email` 索引 |
| `posts` | 预测帖主表 | `id`, `author_id`, `title`, `description`, `source_name`, `status`, `view_count`, `created_at` | `author_id` 外键；`status` 索引；`created_at` 倒序索引 |
| `post_media` | 帖子媒体附件 | `id`, `post_id`, `media_type`, `storage_path`, `public_url`, `file_size`, `created_at` | `post_id` 外键；`media_type` 检查约束 |
| `verification_events` | 验证事件定义 | `id`, `post_id`, `type`, `target_date`, `event_description`, `deadline`, `status`, `triggered_at`, `triggered_by`, `evidence_url` | `post_id` 外键；`type` 与 `status` 索引 |
| `credibility_votes` | 帖子真实度投票 | `id`, `post_id`, `user_id`, `value`, `created_at` | `unique(post_id, user_id)` 防重复投票 |
| `verification_votes` | 验证结果投票 | `id`, `verification_event_id`, `user_id`, `result`, `created_at` | `unique(verification_event_id, user_id)`；`result` 检查约束 |
| `event_trigger_confirms` | 事件触发确认投票 | `id`, `verification_event_id`, `user_id`, `confirmed`, `created_at` | `unique(verification_event_id, user_id)` |
| `comments` | 评论与一级回复 | `id`, `post_id`, `author_id`, `parent_id`, `content`, `score`, `created_at` | `post_id`、`author_id` 外键；`parent_id` 可空；`post_id + created_at` 索引 |
| `notifications` | 站内通知 | `id`, `user_id`, `type`, `reference_id`, `message`, `is_read`, `created_at` | `user_id + is_read` 组合索引 |
| `post_tags` | 帖子标签映射 | `post_id`, `tag_id` | 联合主键 `(post_id, tag_id)` |
| `tags` | 标签字典 | `id`, `name`, `slug` | `slug` 唯一 |

建议补充的数据库规则：

- `posts.status`、`verification_events.type`、`verification_votes.result` 使用数据库枚举或检查约束统一取值
- 对用户写入类操作启用 RLS，至少限制“仅本人可修改自己的资料、帖子补充说明、评论”
- 对投票表强制唯一约束，避免重复投票依赖前端校验
- 媒体文件路径仅保存存储引用，避免将二进制内容写入数据库

### 6.4 MVP 范围说明

- MVP 阶段**不包含支付功能**
- 不接入 PayPal、Stripe 或订阅系统
- 支付、会员、帖子推广等商业化能力放入后续版本评估
- MVP 仅支持 **Google 登录**
- MVP 不引入复杂的用户信誉权重系统，所有投票默认等权

### 6.5 Out of Scope（MVP 不做项）

以下内容明确不进入 MVP 开发范围：

1. 支付、订阅、打赏、广告投放、帖子付费推广

## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 用户上传虚假/伪造内容 | 降低平台公信力 | 真实度投票机制 + 后续增加举报/审核功能 |
| 投票被刷票操纵 | 投票结果失真 | MVP 限制每人一票 + 后续增加信誉权重机制 |
| 版权/肖像权争议 | 法律风险 | 明确用户协议，要求上传内容为公开信息；设置 DMCA 申诉通道 |
| 冷启动期内容不足 | 用户体验差 | 团队自行录入高质量种子内容 |
| 媒体存储成本 | 运营成本压力 | 限制文件大小，视频压缩处理，按需扩容 |
